Goals

Gotchas

  • empty pipe || in the movies file – how to handle that

  • rating IDs autoincrementing every time a seed.py file is run, and ids starting from where they left off (so the first item in the database doesn’t have id=1, it has id=400001…)

  • SQLAlchemy returning Base Query Object because students haven’t added .one, .first or .all to the end of the query.

Code Reviews

Part One

Code Review 1

Students have created a Movie and Ratings table using SQLAlchemy, pattern matching with the User table that was already in the exercise.

They should have psql and python running in separate windows.

There has also been an overview of ORMs, a discussion of the data that we’re using for the exercise, and an outline of the data model.

Students won’t yet have foreign keys.

  • Check what they’ve made nullable to make sure it makes sense with the data model (sometimes students make score nullable in the Rating class)

  • Length of db.String(x) that they’ve given

  • Docstrings might not be updated because they were copied over from the User class example.

Code Review 2

Goes through the rest of Part 1 - adding to the database and reading in seed data.

  • Students have created users in Postgres directly and in Python using SQLAlchemy and have played around with how to get info using both methods.

  • Created __repr__() methods (might get the question about why you don’t need to drop your database to update the __repr__ method on the class)

  • Read the seed data and populate the database. Seed.py contains a rough outline of the process. Load_users is already in the file; they need to write movies and ratings.

  • Discussion of autoincrement. We provide a function that starts autoincrement – do students know what it does? Why would there be an error if we didn’t have the set_val_user_id in the seed.py table?

  • Datetime – students need to use strptime to convert the release date string to a datetime object. The instructions provide sample code to do this.

  • Students should strip off the year from each movie title.

  • It is a good high-level point to talk about that the sqlalchemy classes represent tables in SQL and that the instances of the classes represent rows.

Further Study

Unicode – doesn’t ask for a code review, but gives the students more info on unicode and latin-1.

Part Two

Code Review 1

  • Previously, students read in seed data to the database.

SQL ALCHEMY SECTION:

  • They’ve practiced querying using SQLAlchemy. Questions: what do .all(), .first() and .one() return?

  • As practice, they’re given the question: see if you can figure out how to query users by zipcode and find all users of a particular age. (ex. users that are 17 years old and live in Berwyn, IL).

  • Also have seen order_by and filter_by

  • Using relationships – they’ve now added relationships between movies/ratings and users/ratings. Check in to see if they understand what backref does and where the relationship is actually established in the code. There are two ways of creating relationships; are they familiar with the syntax for both (w/ and w/out backref)?

FLASK SECTION:

Students are given the code to do the following (so ask them about these pieces to make sure they know what they were given)

  • Created a base.html and homepage.html

  • Given a skeleton Flask app.

  • Make a route to display all users and an HTML page to render.

Their tasks:

  • Create a route that renders a sign-in form and a route that adds a user to the database. Told to use sessions.

  • Create a route to handle user login/logout. After a user logs in, they should be redirected to the homepage.

Code Review 2

  • User detail page – students create a page that shows information about a particular user. The user list page should link to the individual user page.

  • Should display age, zipcode and list of movies rated.

  • Login route should now direct user to their individual page.

Code Review 3

  • Students have created a movie list page and a movie detail page. The movie list page should link to each individual movie page.

  • The movie list page should be ordered by title.

  • Movie detail page should have a form for rating the movie and a route to add the rating to the database.

  • The form should also be able to update the rating if the user has already rated it.

Part Three

Doesn’t ask students for a code review, but in this piece, students make the app predictive. The instructions lay out specifically what needs to happen, so students can use code given to them, or they can write their own code.
